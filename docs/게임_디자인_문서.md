# 게임 디자인 문서 — 평민 → 성장 2D 육성 게임

## 요약 3줄

- **평민**이 낚시·채집·채광·장사로 **아이템을 모으고**, 조합·판매로 **성장**하는 웹 2D 육성 게임이다.
- **에너지/레벨/칭호**로 진행을 제어하고, **미니게임**과 **일일 보상**으로 반복 플레이에 변화를 준다.
- **레벨·아이템·스킨**은 데이터(JSON/설정) 추가만으로 확장 가능하도록 설계한다.

---

## 1. 게임 개요

### 1.1 세계관·목표

- **시작**: 이름과 직업 계열(어부/농부/광부/장사꾼)을 고르는 **평민** 한 명.
- **목표**: 아이템 수집·조합·판매로 **레벨업·칭호·골드·명성**을 쌓고, “성공한 직인/상인”으로 성장하는 것.
- **승리 조건**: 명시적인 “엔딩”보다는 **무한 성장 + 도달 목표**(예: 특정 레벨, 칭호, 골드)를 달성하는 형태.  
  - 선택: (A) 무한 성장만 (B) “왕/대부호” 같은 최종 목표 1개 (C) 챕터별 목표(1장: 어부 인정, 2장: 상인 인정…)  
  - **권장**: (A)로 시작해, 나중에 (B)나 (C)를 데이터만 추가해 넣기.

### 1.2 핵심 루프

1. **에너지 소모** → 낚시/채집/채광/장사 중 하나 실행 → **아이템·골드·경험치** 획득  
2. **가방**에서 아이템 확인 → **조합**으로 가공품 제작(소모품·버프·재료)  
3. **시장 등록** → **장사**로 판매 → 골드·명성 획득  
4. **레벨업** 시 스탯 상승, **칭호** 달성 시 보상/이벤트 문구  
5. (선택) **미니게임** 클리어 시 보너스 아이템·에너지

- **에너지**: 일일 제한(예: 100) + 유니크 드랍/미니게임/소모품으로 회복. “노가다량”을 자연스럽게 제한.

### 1.3 직업 계열(어부/농부/광부/장사꾼)

- **역할**: 시작 시 **보너스**만 다르게 준다. (예: 어부 = 낚시 1회당 추가 경험치, 장사꾼 = 장사 골드 +5%)  
- **공통**: 모든 행동(낚시/채집/채광/장사)은 다 할 수 있고, 칭호도 모두 열림.  
- **확장**: 나중에 “전직”이나 “2차 직업”을 넣을 수 있도록 **characterType**만 저장해 두고, 보너스 수식만 바꾸면 됨.

---

## 2. 조작 방식

### 2.1 플랫폼

- **웹**: PC/모바일 브라우저. 터치·클릭·키보드(선택) 모두 고려.
- **UI**:  
  - (A) **채팅형**: 명령어 입력 + 버튼(퀵리플라이). 구현 단순, 확장 쉬움.  
  - (B) **버튼·카드형**: 메뉴/행동을 카드·버튼으로만 선택. 직관적이지만 화면 수 많아짐.  
  - (C) **하이브리드**: 메인은 버튼, 상세(조합 번호, 사용할 아이템 등)만 텍스트 입력.  
  - **권장**: (A) 또는 (C). “명령어 한 줄 + 버튼”만 있어도 1차 구현은 충분.

### 2.2 기본 입력

- **캐릭터 생성**: 이름(2~6 한글) + 직업 계열 선택(어부/농부/광부/장사꾼).
- **행동**: “낚시” “채집” “채광” “장사” — 1회 실행 시 에너지 1 소모.
- **정보**: “메뉴”(상태 요약), “내 정보”(상세), “가방”, “시장 등록”, “시장 현황”.
- **조합**: “조합”(가능 목록), “조합 1”(1번 레시피 제작).
- **사용**: “사용 [가공품 이름]”(소모품/버프 사용).

---

## 3. 승리/패배 조건

### 3.1 승리(목표 달성)

- **1차**: “성장 게임”이므로 **명시적 게임오버 없음**. 목표는 “레벨 N 달성”, “칭호 O 달성”, “골드 M 모으기” 등.
- **선택적 엔딩**:  
  - (A) 없음: 무한 성장만.  
  - (B) 하나: 예) “명성 10000 + 레벨 50” 달성 시 “대부호 엔딩” 문구 + 계속 플레이 가능.  
  - (C) 여러 개: 칭호별/레벨별 짧은 “도달 메시지”만.  
  - **권장**: (A)로 시작, (C)로 칭호 달성 시 축하 문구만 넣기. (B)는 콘텐츠가 쌓인 뒤 추가.

### 3.2 패배(게임오버)

- **기본**: 게임오버 없음. 에너지 0이면 행동만 못 할 뿐, 시간/소모품으로 회복.
- **(선택) 일일 제한**: “오늘 N회만 행동 가능” 등으로 제한을 두고, 다음 날 리셋. 구현은 단순하나 “기다리기” 요소 생김.

---

## 4. 난이도 설계

### 4.1 원칙

- **처음**: 에너지 넉넉, 드랍 풍부, 조합 레시피 적고 재료 단순. “눈에 보이는 성장”이 빠르게 느껴지게.
- **중반**: 레시피·아이템 종류 증가, “시장 등록 → 장사” 루프가 돈 버는 주 수단이 되도록.
- **후반**: 희귀/유니크 비율은 유지하되, “다음 칭호/레벨”까지 요구 경험치·횟수를 서서히 올려서 **노가다량만** 늘린다. 새 규칙을 많이 넣지 않음.

### 4.2 구체 예시

| 구간        | 레벨/칭호 예시      | 에너지/회복        | 아이템/조합                    |
|------------|---------------------|--------------------|-------------------------------|
| 초반       | 1~10, 무명→후보생   | 100/일, 회복 쉬움  | 보통 위주, 레시피 3~5개       |
| 중반       | 10~30, 초짜→경력직 | 동일 또는 소폭 상승| 레어 비율 소폭 상승, 레시피 7~10개 |
| 후반       | 30~50+, 명인~전설   | 동일               | 유니크 확률 유지, “횟수”만 많이 요구 |

- **난이도 올리는 방법**: (1) 칭호 구간별 필요 횟수만 증가 (2) 상위 레시피 재료 개수만 증가 (3) 새 아이템/레시피 추가.  
- **피할 것**: “즉사”, “아이템 파괴”, “랜덤 패널티” 같은 강한 부정 요소. 실패는 “재료 부족으로 조합 불가” 수준으로만.

### 4.3 랜덤성

- **드랍**: 보통/레어/유니크 비율을 **고정 확률**(예: 81% / 17% / 2%)으로 두고, “오늘 운이 없음”이 길게 이어지지 않게.
- **장사 가격**: 80~120% 같은 **좁은 변동**만. 큰 폭 변동은 (구현 난이도 높음) 밸런스도 어려우니 1차에서는 생략 권장.

---

## 5. 추가 아이디어

### 5.1 미니게임 (확장)

- **역할**: 에너지 회복, 특수 아이템, 재미 요소. **메인 루프와 분리**된 짧은 게임.
- **예시**:  
  - (A) **숫자 맞추기**: 1~10 랜덤, 3번 안에 맞추면 에너지 +5. 구현 매우 쉬움.  
  - (B) **가위바위보 3판**: NPC와 대전, 2승 시 보상. 구현 쉬움.  
  - (C) **간단 슬라이더/타이밍**: 바가 특정 구간에 들어가면 성공. (구현 난이도 중)  
- **권장**: (A) 또는 (B)를 먼저 하나 넣고, “미니게임 입장 → 성공/실패 → 보상” 흐름만 확립한 뒤 종류만 추가.

### 5.2 일일 보상 / 출석

- **아이디어**: 접속일 수에 따라 “연속 N일 출석 보상”(골드, 소모품, 에너지).  
- **구현**: `lastLoginDate`, `loginStreak` 저장 후, 날짜가 바뀌었으면 streak 갱신 + 보상 지급. (구현 난이도 낮음)

### 5.3 스킨 / 외형

- **아이디어**: 칭호·레벨·도달 목표에 따라 **캐릭터/카드/버튼** 이미지만 바꾸기.  
- **구현**: `characterType` + `skinId` 같은 값을 두고, 프론트에서 이미지/클래스만 매핑. (구현 난이도 낮음, 리소스 제작은 별도)

### 5.4 레벨/아이템 확장

- **레벨**: “필요 경험치 = 100 * 레벨” 같은 **단순 수식**으로 두고, 레벨 캡만 데이터로 조정.  
- **아이템/레시피**: JSON 배열(아이템 정의, 드랍테이블, 레시피)에 항목만 추가. 코드는 “타입/등급”으로만 분기하도록 설계.

---

## 6. 구현 시 우선순위 제안

1. **1차**: 캐릭터 생성(이름+직업) → 에너지·레벨·칭호·낚시/채집/채광/장사 → 가방·조합·시장 등록·장사 → 웹 UI(채팅형 또는 버튼형).  
2. **2차**: 미니게임 1종, 일일 에너지 리셋, (선택) 일일 보상.  
3. **3차**: 스킨/외형, 추가 레시피·아이템, (선택) 엔딩/도달 목표.

---

## 7. 표기 규칙 (문서 내)

- **(구현 난이도 높음)**: 규칙/기능이 복잡하거나 밸런스·예외 처리 부담이 큰 경우.  
- **권장**: 여러 선택지 중에서 1차 구현에 추천하는 방향.

이 문서는 “코드 없이” 기획·룰만 정리한 것이므로, 구현 시 `gameEngine`·`state`·`config` 구조와 맞춰 가면 된다.
